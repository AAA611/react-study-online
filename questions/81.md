# React 输出题 81：外层 Fragment 的 key 变化会让整棵子树重新挂载

## 基本信息
- 分类：渲染模型 / diff 算法
- 难度：中级
- 标签：Fragment, key, 重新挂载
- 题型：阅读 React 代码并写出输出

## 题目描述
虽然 `Child` 本身没有变化，但外层 `Fragment` 的 `key` 变了。控制台输出顺序是什么？

## 代码
```jsx
function Child() {
  const [count] = React.useState(1);

  React.useEffect(() => {
    console.log("mount");

    return () => {
      console.log("unmount");
    };
  }, []);

  console.log("child", count);
  return null;
}

function App() {
  const [flag, setFlag] = React.useState(false);

  React.useEffect(() => {
    setFlag(true);
  }, []);

  return (
    <React.Fragment key={flag ? "B" : "A"}>
      <Child />
    </React.Fragment>
  );
}

render(<App />);
```

## 参考答案
输出顺序是：`child 1 -> mount -> child 1 -> unmount -> mount`。

## 考察点
- 外层节点的 `key` 变化会让整棵子树失去复用条件
- React diff 看的是整棵树的身份，不只是最内层组件
- 子树被替换时会触发卸载和重新挂载

## 详细讲解
第一次渲染时，`Child` 初始化并输出 `child 1`，提交后执行 effect，输出 `mount`。当 `flag` 变成 `true` 后，虽然 `Child` 还是那个 `Child`，但它外面的 `Fragment` 的 key 从 `"A"` 变成了 `"B"`。对于 React 来说，这已经不是同一棵子树了，所以旧子树会被卸载，新子树重新挂载。于是会再次渲染出一个新的 `Child`，输出第二次 `child 1`，然后旧实例 cleanup 输出 `unmount`，新实例再输出 `mount`。
